# Backend Verification Guide for Teammates

## Purpose

This guide explains how to manually verify the current CodeDNA backend from a clean local setup.

It is written for someone who is new to the project or new to backend systems.
It explains what each step proves, what terms mean, and what a correct result looks like.

The current backend supports:

- FastAPI application startup.
- PostgreSQL database migrations.
- Redis connectivity.
- GitHub OAuth login.
- Local CodeDNA JWT authentication.
- GitHub repository discovery.
- Repository import into CodeDNA.
- Asynchronous indexing scaffold using Celery and Redis.
- Job status tracking in PostgreSQL.

The current backend does not yet perform real repository cloning, parsing, embeddings, graph generation, or AI analysis.
The indexing task is intentionally a stub.

## Required Local Setup

You need:

- Docker Desktop or Docker Engine running.
- GitHub OAuth app configured.
- `apps/api/.env` populated with local development values.
- Port `8001` available, or another API port of your choice.

The local API URL used in this guide is:

```text
http://localhost:8001
```

If you use the default Compose port instead, replace `8001` with `8000`.

## Environment Variables

The backend reads configuration from environment variables through Pydantic Settings.

For local development, create `apps/api/.env` using `apps/api/.env.example` as the template.

Important values:

| Variable | What it means |
| --- | --- |
| `DATABASE_URL` | Connection string for PostgreSQL. |
| `REDIS_URL` | Connection string for Redis. |
| `GITHUB_CLIENT_ID` | Public identifier for the GitHub OAuth app. |
| `GITHUB_CLIENT_SECRET` | Secret used by the backend to exchange GitHub OAuth codes. |
| `GITHUB_CALLBACK_URL` | URL GitHub redirects back to after approval. |
| `JWT_SECRET` | Secret used by CodeDNA to sign its own JWTs. |
| `JWT_ALGORITHM` | Signing algorithm, usually `HS256` locally. |
| `JWT_EXPIRE_MINUTES` | How long a CodeDNA login token is valid. |

Example GitHub callback URL for local development:

```text
http://localhost:8001/api/v1/auth/github/callback
```

This exact URL must also be configured in the GitHub OAuth app settings.

## Start the Backend

Run:

```bash
API_PORT=8001 docker compose up -d --build api postgres redis worker
```

Then apply database migrations:

```bash
docker compose run --rm --build migrate
```

What this does:

- `api` starts the FastAPI server.
- `postgres` starts the database.
- `redis` starts the message broker.
- `worker` starts the Celery background worker.
- `migrate` applies Alembic database migrations.

Check services:

```bash
docker compose ps
```

Expected result:

- `codna-api` should be `healthy`.
- `codna-postgres` should be `healthy`.
- `codna-redis` should be `healthy`.
- `codna-worker` should be `Up`.

## Check Basic Health

Open:

```text
http://localhost:8001/api/v1/health
```

Expected response:

```json
{
  "status": "ok"
}
```

Also check:

```text
http://localhost:8001/api/v1/live
http://localhost:8001/api/v1/ready
```

What this proves:

- The API process is running.
- FastAPI routing works.
- Readiness confirms dependent services such as database and Redis are reachable.

## Use Swagger

Open:

```text
http://localhost:8001/docs
```

Swagger is the interactive API page generated by FastAPI.

Think of Swagger as a browser-based API tester.
Instead of writing `curl` commands, you can click an endpoint, fill input fields, and press `Execute`.

## Step 1: Start GitHub Login

In Swagger, run:

```http
GET /api/v1/auth/github/login
```

Expected response contains an `authorization_url`.

Open that `authorization_url` in the browser.

What this does:

- Sends you to GitHub.
- GitHub asks whether you allow the CodeDNA OAuth app to read your account/repository metadata.
- After approval, GitHub redirects back to the backend callback URL.

Important:

- This flow does not allow CodeDNA to push code or modify repositories.
- Current backend usage is read-only metadata access.
- Do not paste GitHub access tokens into chat or commit them.

## Step 2: Copy the CodeDNA Access Token

After GitHub redirects back, the backend returns a response like:

```json
{
  "access_token": "eyJhbGciOiJIUzI1NiIs...",
  "token_type": "bearer",
  "expires_in": 3600,
  "user": {
    "id": "13834d99-e025-48fe-a640-067f600bb9a2",
    "github_id": "12345",
    "username": "example-user",
    "email": "example@example.com",
    "name": "Example User",
    "avatar_url": "https://avatars.githubusercontent.com/u/12345?v=4"
  }
}
```

Copy only the `access_token`.

This token is a CodeDNA JWT, not the GitHub access token.

The GitHub access token stays backend-only. The frontend and Swagger use only the CodeDNA JWT.

## Step 3: Authorize Swagger

In Swagger:

1. Click `Authorize`.
2. Paste:

```text
Bearer <your-codedna-access-token>
```

Example:

```text
Bearer eyJhbGciOiJIUzI1NiIs...
```

3. Click `Authorize`.
4. Close the modal.

Common mistake:

Do not paste only the token without `Bearer`.

Correct:

```text
Bearer eyJhbGciOiJIUzI1NiIs...
```

Incorrect:

```text
eyJhbGciOiJIUzI1NiIs...
```

## Step 6: List GitHub Repositories

Run:

```http
GET /api/v1/github/repositories
```

Use these query parameters:

```text
visibility = all
sort = updated
page = 1
per_page = 30
```

Expected result:

```json
{
  "repositories": [
    {
      "github_id": "123456789",
      "name": "example-repo",
      "full_name": "example-user/example-repo",
      "default_branch": "main",
      "clone_url": "https://github.com/example-user/example-repo.git",
      "visibility": "private",
      "private": true
    }
  ],
  "page": 1,
  "per_page": 30,
  "has_next_page": false
}
```

What this proves:

- The backend can ask GitHub for repositories accessible to the user.
- Pagination works.
- GitHub metadata is normalized into CodeDNA response schemas.

Pick one repository from the response and copy either:

- `full_name`, for example `example-user/example-repo`.
- `github_id`, for example `123456789`.

For manual testing, `full_name` is easier to read.

## Step 7: Import a Repository into CodeDNA

Run:

```http
POST /api/v1/repositories
```

Request body:

```json
{
  "full_name": "example-user/example-repo"
}
```

Expected response:

```json
{
  "name": "example-repo",
  "full_name": "example-user/example-repo",
  "default_branch": "main",
  "clone_url": "https://github.com/example-user/example-repo.git",
  "visibility": "private",
  "status": "registered",
  "id": "2353ab83-0db6-49e4-8503-83fd1fc0bfef",
  "owner_id": "13834d99-e025-48fe-a640-067f600bb9a2",
  "github_id": "123456789",
  "last_indexed_at": null,
  "created_at": "2026-07-17T21:29:03.856134Z",
  "updated_at": "2026-07-17T21:29:03.856134Z"
}
```

Copy the returned `id`. This is the local CodeDNA repository ID.

Example:

```text
repository_id = 2353ab83-0db6-49e4-8503-83fd1fc0bfef
```

What this proves:

- The frontend does not send trusted metadata such as `clone_url`, `visibility`, or `default_branch`.
- The backend fetches canonical metadata from GitHub.
- The repository is stored in PostgreSQL.
- Ownership is tied to the local CodeDNA user.

Common mistake:

Do not use the GitHub `github_id` as the path parameter for CodeDNA repository routes.
After import, CodeDNA routes use the local `id` returned by `POST /api/v1/repositories`.

## Step 10: Start the Indexing Stub

Run:

```http
POST /api/v1/repositories/{repository_id}/index
```

Example:

```http
POST /api/v1/repositories/2353ab83-0db6-49e4-8503-83fd1fc0bfef/index
```

Expected response:

```json
{
  "repository_id": "2353ab83-0db6-49e4-8503-83fd1fc0bfef",
  "job_id": "87bfc7b3-278d-4385-a5cf-80656e0065e9",
  "status": "queued"
}
```

Copy the returned `job_id`.

What this proves:

- The API created a job row in PostgreSQL.
- The API published a Celery task to Redis.
- The API returned immediately instead of waiting for the worker.

Common mistake:

Do not use Swagger's example UUID:

```text
3fa85f64-5717-4562-b3fc-2c963f66afa6
```

That is only placeholder data. Use the real `job_id` returned by the indexing endpoint.

## Step 11: Check Job Status

Run:

```http
GET /api/v1/jobs/{job_id}
```

Example:

```http
GET /api/v1/jobs/87bfc7b3-278d-4385-a5cf-80656e0065e9
```

Immediately after enqueueing, the status may be:

```json
{
  "status": "queued"
}
```

or:

```json
{
  "status": "running"
}
```

After one or two seconds, run the same request again.

Expected final result:

```json
{
  "id": "87bfc7b3-278d-4385-a5cf-80656e0065e9",
  "repository_id": "2353ab83-0db6-49e4-8503-83fd1fc0bfef",
  "job_type": "repository_index",
  "status": "completed",
  "celery_task_id": "9b2ab317-9f6b-4d8d-93b5-1925fd62e70e",
  "error_message": null,
  "started_at": "2026-07-17T21:20:40.584017Z",
  "completed_at": "2026-07-17T21:20:41.624748Z",
  "created_at": "2026-07-17T21:20:40.484369Z",
  "updated_at": "2026-07-17T21:20:41.617997Z"
}
```

What this proves:

- The worker consumed the task.
- The worker updated the job from `queued` to `running` to `completed`.
- The database is the source of truth for job status.

## Step 12: Confirm Repository Status Updated

## Step 14: Check the Database Directly

Repositories:

```bash
docker compose exec postgres psql -U postgres -d codna -c \
  "select id, owner_id, full_name, status, last_indexed_at from repositories order by created_at desc limit 5;"
```

Jobs:

```bash
docker compose exec postgres psql -U postgres -d codna -c "
select id, repository_id, job_type, status, celery_task_id,
       error_message, created_at, completed_at
from jobs
order by created_at desc
limit 5;
"
```

Migrations:

```bash
docker compose exec postgres psql -U postgres -d codna -c "select version_num from alembic_version;"
```

Expected migration version:

```text
20260717_000006
```

What this proves:

- Migrations are applied.
- Repository records exist.
- Job records exist.
- Job state is persisted outside Redis.

## Beginner Term Glossary

### API

An API is how one program talks to another program.

Example:

The frontend asks:

```text
GET /api/v1/repositories
```

The backend answers with JSON:

```json
[
  {
    "full_name": "example-user/example-repo"
  }
]
```

### FastAPI

FastAPI is the Python framework running the backend HTTP server.

In this project, FastAPI receives requests such as:

- Login with GitHub.
- List GitHub repositories.
- Import repository.
- Start indexing.
- Read job status.

### Endpoint

An endpoint is one specific API URL and method.

Example:

```http
POST /api/v1/repositories/{repository_id}/index
```

This endpoint starts an indexing job for one repository.

### HTTP Method

The method describes what kind of action the request is doing.

Common methods:

| Method | Meaning | Example |
| --- | --- | --- |
| `GET` | Read data | Get repository list. |
| `POST` | Create or start something | Import repository or start indexing. |

### JSON

JSON is the data format used in API requests and responses.

Example:

```json
{
  "full_name": "example-user/example-repo"
}
```

### GitHub OAuth

OAuth is a secure login/authorization flow.

Instead of giving CodeDNA your GitHub password, you approve access on GitHub. GitHub then gives the backend a temporary access token.

Example flow:

```text
User clicks login
        |
        v
GitHub approval page
        |
        v
GitHub redirects back to CodeDNA
        |
        v
CodeDNA creates its own JWT
```

### GitHub Access Token

This is the token GitHub gives to the backend after OAuth.

The backend uses it to call GitHub APIs, such as:

- Read GitHub profile.
- List repositories.
- Fetch repository metadata.

The GitHub access token should stay backend-only.

### CodeDNA JWT

JWT means JSON Web Token.

The CodeDNA JWT is the token the backend gives to the frontend after GitHub login.

The frontend sends it like this:

```text
Authorization: Bearer <codedna-jwt>
```

The backend uses it to know which local CodeDNA user is making the request.

### Bearer Token

`Bearer` is the authentication scheme used in the HTTP header.

Correct header:

```text
Authorization: Bearer eyJhbGciOiJIUzI1NiIs...
```

It means:

```text
Here is my token. Use it to authenticate me.
```

### PostgreSQL

PostgreSQL is the main database.

In this project, it stores durable records such as:

- Users.
- Imported repositories.
- Jobs.

Durable means the data remains even if the API process restarts.

### Migration

A migration is a versioned database change.

Example:

The jobs milestone added a migration that creates the `jobs` table.

Why migrations matter:

- Every teammate gets the same database shape.
- Production can be upgraded safely.
- Database changes are tracked in Git.

### Alembic

Alembic is the migration tool used with SQLAlchemy.

Command:

```bash
docker compose run --rm --build migrate
```

This applies pending migrations to PostgreSQL.

### SQLAlchemy

SQLAlchemy is the Python database layer.

It lets the backend work with Python classes instead of writing raw SQL everywhere.

Example concept:

```text
Repository model -> repositories table
Job model -> jobs table
User model -> users table
```

### Redis

Redis is an in-memory service used here as the Celery message broker.

In this project, Redis does not store permanent business data. It carries background task messages from the API to the worker.

Example:

```text
API says: run job 123
Redis stores the message briefly
Worker picks it up
```

### Celery

Celery is the background job system.

FastAPI should stay fast and avoid doing slow work inside HTTP requests.
Celery lets the backend send slow work to a separate worker process.

Example:

```text
POST /repositories/{id}/index
        |
        v
Create job row
        |
        v
Send task to Celery
        |
        v
Return response immediately
```

### Worker

A worker is a separate process that performs background tasks.

In this project, the worker currently runs a stub indexing task.

Later, the worker will handle:

- Repository cloning.
- Parsing source code.
- Creating chunks.
- Creating embeddings.
- Building graph records.

### Job

A job is a durable record that tracks background work.

Example job statuses:

```text
queued -> running -> completed
```

If something fails:

```text
queued -> running -> failed
```

The frontend should read job status from:

```http
GET /api/v1/jobs/{job_id}
```

### Repository Import

Repository import means registering a GitHub repository inside CodeDNA.

It does not mean cloning yet.

Current import stores metadata:

- GitHub ID.
- Name.
- Full name.
- Default branch.
- Clone URL.
- Visibility.
- Owner.
- Status.

### Indexing

Indexing means preparing a repository so CodeDNA can search, understand, and answer questions about it.

Current milestone:

- Indexing is a stub.
- It only tests background infrastructure.

Future milestone:

- Clone repository.
- Parse files.
- Extract symbols.
- Create chunks.
- Store embeddings.
- Build graph records.

### Stub

A stub is placeholder logic that proves the wiring works before real logic is added.

Current stub task:

```text
mark job running
wait briefly
mark job completed
mark repository ready
```

This proves the async architecture works before adding expensive indexing logic.

## Common Problems and Fixes

### `401 Authentication required`

Cause:

- Swagger is not authorized.
- Token is missing.
- Token was pasted without `Bearer`.

Fix:

Authorize Swagger with:

```text
Bearer <codedna-jwt>
```

### `401 Invalid authentication token`

Cause:

- Token expired.
- Token was copied partially.
- Wrong token was used.
- Backend restarted with a different `JWT_SECRET`.

Fix:

Run GitHub login again and copy a fresh CodeDNA `access_token`.

### `404 Job not found`

Cause:

- You used Swagger's placeholder UUID.
- You used a repository ID instead of a job ID.
- You are logged in as a different user from the job owner.

Fix:

Call:

```http
POST /api/v1/repositories/{repository_id}/index
```

Then copy the returned `job_id` and use it here:

```http
GET /api/v1/jobs/{job_id}
```

### `404 Repository not found`

Cause:

- Repository was not imported yet.
- You used GitHub `github_id` instead of local CodeDNA repository `id`.
- The repository belongs to a different local user.

Fix:

Call:

```http
GET /api/v1/repositories
```

Use the `id` from that response.

### `409 Conflict`

Cause:

- The same user already imported that GitHub repository.

Fix:

Use:

```http
GET /api/v1/repositories
```

Then copy the existing repository `id`.

### `422 Unprocessable Entity`

Cause:

- Request body is malformed.
- Both `github_id` and `full_name` were sent.
- Neither `github_id` nor `full_name` was sent.
- Frontend metadata such as `clone_url` or `status` was sent.

Correct body:

```json
{
  "full_name": "example-user/example-repo"
}
```

or:

```json
{
  "github_id": "123456789"
}
```

### Worker does not process jobs

Check:

```bash
docker compose ps worker
docker compose logs --tail=100 worker
```

Expected:

```text
celery ready
Task app.workers.tasks.run_repository_index received
Task app.workers.tasks.run_repository_index succeeded
```

If the worker is not running:

```bash
API_PORT=8001 docker compose up -d --build worker
```

### Migration missing jobs table

Check:

```bash
docker compose exec postgres psql -U postgres -d codna -c "select version_num from alembic_version;"
```

Expected:

```text
20260717_000006
```

If not:

```bash
docker compose run --rm --build migrate
```

## What a Successful Full Verification Proves

If every step passes, the current backend foundation is working:

- FastAPI starts correctly.
- Docker services communicate.
- Pydantic Settings loads environment configuration.
- PostgreSQL migrations are applied.
- GitHub OAuth creates a local user.
- CodeDNA JWT authentication works.
- GitHub metadata can be read.
- Repository registration persists canonical metadata.
- Celery can enqueue and execute background work.
- Redis works as the broker.
- Job status is persisted and readable.
- The public API is ready for the real indexing milestone.

## What This Does Not Prove Yet

This guide does not verify:

- Repository cloning.
- GitHub sync.
- Tree-sitter parsing.
- Semantic chunking.
- Embeddings.
- pgvector search.
- Knowledge graph construction.
- AI responses.

Those are future milestones.
